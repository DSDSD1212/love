# 技术调研 (Research): 动态图片加载

## 面临的挑战：在没有后端的情况下动态加载图片

功能需求说明指出，要把所有存放在 `assets/images/` 目录里的图片都加载出来，而不是把图片名字写死在代码里。但与之矛盾的是，系统技术架构约束了一个仅使用原生前端技术（Vanilla Frontend）的 Web 应用，且不存在任何后端服务。

在标准运行于网页浏览器的原生 JavaScript 方案中，受安全限制，前端通常是无法访问本地文件系统或去读取服务器某个目录里面的内容（目录列表）的。

## 决策：是构建时生成 JSON，还是去获取一个预定义的列表？

既然纯前端 JS 无法列出目录内容，那么我们就必须通过某种手段把文件列表直接提供给前端。
这里有几种可能的方案：
1. **[否决 REJECTED]** 采用后端服务（比如 Node.js, PHP 等）去读取和返回目录内容。（不符合无后端服务架构这一硬性约束）。
2. **[否决 REJECTED]** 加入构建环节/工具链（比如 Webpack, Vite）以便在构建期间注入这个文件列表。（目前项目是个简单的、连构建流程都没有的纯单体 HTML 文件，这样做太重了）。
3. **[采纳 SELECTED]** 开发一个简单的自动化脚本（比如 shell/bash 或者需要手动执行的 Node.js 脚本），由它在 `assets/images/` 目录下生成一个包含所有图片信息的配置文件（如 `config.json` 或 `images.json`）。然后前端的 JS 再利用 `fetch()` 去读取这个简单的 JSON 即可。

### 采用该方案的理由 (Rationale)

使用脚本生成一个 JSON 静态文件是所有方案里最稳妥、也最极致简单的做法，同时也能完美契合现有的纯前端架构要求。今后如果用户或者 CI/CD 流程新加入了别的图片，只需跑一下那个简单的脚本即可实现 JSON 配置文件的更新。这样既不需要后端服务器一直在后台跑着，同时也避免了在一个轻便的项目里引入各种复杂的构建流。

## 其他曾考虑的替代方案 (Alternatives Considered)

另一种做法是直接让前端不断去“猜测”图片名字（像 image1.jpg, image2.jpg 这样挨个试），直到报 404 错误为止。但这严重依赖于图片要有严格命名的前置条件，并且加载效率低，更不支持去拉取各种动态命名甚至带了时间戳等毫无规律的图片名（拿 `cute_love_candies_1772443765380.webp` 做例子）。

## 解决需要澄清的问题 (Resolution of NEEDS CLARIFICATION)

- **所需澄清的点:** "如果纯前端 JS 无法在没有后端支持或无构建步骤的情况下读取服务器目录列表，我们在架构思路上**需要澄清 (NEEDS CLARIFICATION)**。"
- **最终解决:** 我们会使用一段单独的辅助脚本提前产生一个静态 `images.json` 索引，让前端依靠 fetch 去拉取它。这样既保证网页是静态托管的形式，也满足了动态性的要求。
